Flujo de Git para Course Catalog API

Crea el repo · trabaja con ramas · sube solo el código funcional

🔧 Fase 1 — Configuración inicial (se hace UNA sola vez)
1
Crear repositorio en GitHub
Una vez

Ve a github.com → New Repository → Llámalo course-catalog-api → Selecciona Private (o Public) → NO agregues README ni .gitignore (ya los tenemos).

2
Inicializar git en la carpeta local
Una vez

Abre PowerShell en la carpeta back_anajuli y ejecuta:

# Inicializa el repositorio git
git init

# Configura la rama principal como "main"
git branch -M main

# Conecta con GitHub (reemplaza USER con tu usuario)
git remote add origin https://github.com/USER/course-catalog-api.git
3
Primer commit — código funcional base
Una vez

El .gitignore ya excluye los HTMLs. Este commit sube solo el proyecto funcional.

git add .
git commit -m "chore: initial project setup - Gradle + Spring Boot 3.2.5"
git push -u origin main
4
Crear rama develop
Una vez

develop es la rama de integración donde se juntan todas las features antes de ir a main.

git checkout -b develop
git push -u origin develop
🔄 Fase 2 — Flujo diario (cada nueva funcionalidad)
5
Crear rama feature desde develop
Siempre

Cada funcionalidad nueva va en su propia rama. Nunca trabajes directo en main o develop.

# Asegúrate de estar en develop y actualizado
git checkout develop
git pull origin develop

# Crea la rama de feature (ej: dominio, infraestructura, cors, etc.)
git checkout -b feature/domain-model
# Otros ejemplos:
# feature/course-use-case
# feature/in-memory-repository
# feature/rest-controller
# feature/cors-configuration
6
Trabajar y hacer commits en la feature
Siempre

Trabaja normalmente. Usa mensajes de commit con prefijos convencionales:

git add .
git commit -m "feat: add Course domain entity with Builder pattern"

# Prefijos convencionales:
# feat:   nueva funcionalidad
# fix:    corrección de bug
# refactor: refactorización sin cambio de comportamiento
# chore:  configuración, dependencias, .gitignore
# docs:   solo documentación (NO subimos los HTML)
7
Subir la rama y mergear a develop
Siempre

Sube la rama y crea un Pull Request en GitHub hacia develop. O hazlo directo por terminal:

# Subir la rama feature
git push origin feature/domain-model

# Mergear a develop localmente (si no usas PR en GitHub)
git checkout develop
git merge --no-ff feature/domain-model
git push origin develop

# Borrar la rama feature (ya fue mergeada)
git branch -d feature/domain-model
🚀 Fase 3 — Release (cuando el proyecto está completo)
8
Mergear develop → main + crear tag de versión
Al finalizar

Cuando todas las features están en develop y probadas, se fusionan a main y se etiqueta la versión.

# Ir a main y mergear develop
git checkout main
git merge --no-ff develop

# Crear tag de versión
git tag -a v1.0.0 -m "release: v1.0.0 - Course Catalog API completo"

# Subir main y el tag
git push origin main
git push origin --tags

✅ Verificar qué archivos subirán antes de hacer push
💡
Comando de verificación
Tip

Antes de hacer commit, verifica qué archivos va a rastrear git:

# Ver qué archivos están siendo rastreados (deben ser solo código)
git status

# Ver exactamente qué archivos git ESTÁ rastreando (excluidos los ignorados)
git ls-files

# Verificar que un archivo específico está ignorado
git check-ignore -v CODIGO_Y_LOGICA.html
