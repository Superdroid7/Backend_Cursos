📋 Discurso del Proyecto

Course Catalog API

Entiende qué hace, por qué existe y cómo se construyó — en menos de 5 minutos.

A · Cómo funciona

¿Qué pasa cuando el frontend pide cursos?

El frontend Angular hace una petición HTTP GET a http://localhost:8080/api/courses. El backend la recibe, la procesa y responde con una lista de cursos en formato JSON.

El flujo interno es sencillo: la petición entra al CourseController, que toma los parámetros opcionales (search y category), los pasa al CourseUseCase (la interfaz del dominio), y el CourseService aplica la lógica de filtrado combinando ambos criterios. Finalmente, el resultado se convierte en DTOs (CourseResponse) y se serializa como JSON.

Adicionalmente, se ha integrado la funcionalidad de creación. Mediante una petición POST se puede enviar un curso nuevo (con título, descripción, categoría, temas/lenguajes, URL de imagen y un enlace). Esto pasa por un DTO (CourseRequest), es validado por el dominio y almacenado en el repositorio.

En una frase: "El controller recibe, el servicio piensa, el repositorio guarda o busca, el DTO empaqueta y la respuesta viaja."

Todo esto está documentado interactivamente con Swagger (Springdoc OpenAPI) en la ruta /swagger-ui.html. El sistema soporta estos endpoints:

GET
/api/courses
Todos los cursos sin filtro
GET
/api/courses?search=python
Búsqueda por palabra clave en título o descripción
GET
/api/courses?category=CLOUD
Filtro por categoría (PROGRAMMING, DESIGN, CLOUD, TESTING)
GET
/api/courses?search=docker&category=PROGRAMMING
Búsqueda combinada: keyword + categoría
GET
/api/courses/{id}
Curso específico por ID (ej: /api/courses/3)
POST
/api/courses
Creación de un nuevo curso con temas (themes) y enlace (link)
B · Por qué se hizo y su propósito

El problema que resuelve

El frontend del catálogo de cursos existía solo como HTML y CSS estáticos. Mostraba tarjetas de cursos hardcodeadas directamente en el markup — sin búsqueda real, sin filtros, sin datos dinámicos.

El backend nació para darle vida a ese frontend. Ahora, en lugar de HTML estático, Angular puede consumir una API REST y mostrar los cursos dinámicamente, filtrando por categoría ("Programming", "Cloud", "Testing", "Diseño") o buscando por nombre.

Propósito técnico: Demostrar arquitectura hexagonal en un proyecto Java Spring Boot real, con separación estricta de capas, sin base de datos (datos en memoria para facilitar el desarrollo), y listo para escalar a PostgreSQL en el futuro sin cambiar una línea del dominio.

Usar arquitectura hexagonal significa que el negocio (qué es un curso, qué se puede buscar) vive aislado del resto. Si mañana el cliente decide cambiar la base de datos, cambiar el framework web, o agregar una interfaz CLI — el dominio no se toca.

C · Orden de construcción

Cómo se armó el proyecto paso a paso

Paso 1 — Proyecto base
build.gradle + settings.gradle + application.properties
Se configuró Gradle con Groovy DSL, Spring Boot 3.2.5, Java 17 y Lombok. application.properties solo define el puerto (8080) — sin base de datos.
Paso 2 — Dominio
CourseCategory.java + Course.java
Se definieron los objetos de negocio. El enum garantiza categorías válidas. La entidad Course incluye título, descripción, categoría, URL de imagen, temas cubiertos (themes) y enlace al curso (link). Es inmutable y se construye con Builder.
Paso 3 — Puertos
CourseUseCase.java + CourseRepositoryPort.java
Los contratos del dominio. Dejan claro qué puede hacer el sistema (UseCase) y qué necesita del almacenamiento (RepositoryPort), sin comprometerse con ninguna tecnología.
Paso 4 — Lógica de negocio
CourseService.java
La inteligencia del sistema. Implementa el UseCase: valida entradas, combina filtros de keyword y categoría, y delega la búsqueda al repositorio.
Paso 5 — Datos en memoria
InMemoryCourseRepository.java
El adaptador de salida. Contiene la lista estática de 6 cursos (Learn Python, HTML/CSS, JavaScript, Docker, AWS, JUnit). Implementa CourseRepositoryPort con streams de Java.
Paso 6 — API REST y Swagger
CourseResponse/Request.java + CourseController.java
Los DTOs definen el JSON de petición/respuesta. El controller expone los endpoints REST. Además, está integrado con Springdoc OpenAPI, generando documentación visual automática accesible en /swagger-ui.html.
Paso 7 — Configuración
BeanConfiguration.java + CorsConfig.java
BeanConfiguration ensambla el repositorio + servicio sin contaminar el dominio con @Service. CorsConfig permite que Angular en localhost:4200 consuma la API sin bloqueos CORS.
