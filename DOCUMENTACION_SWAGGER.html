📜 Swagger y Arquitectura Hexagonal

El manual definitivo para documentar tu API automáticamente sin romper las reglas de Clean Architecture.

🛡️ 1. Mantenimiento de la Arquitectura Hexagonal

¡No te preocupes! La Arquitectura Hexagonal se mantiene completamente intacta y purista.

En esta arquitectura, la regla de oro es que el Dominio (el corazón del hexágono) no puede conocer NADA sobre la infraestructura (Base de datos, Frameworks Web, o Librerías de Documentación).

  • Course, CourseUseCase y CourseService siguen sin tener ni una sola anotación de Spring, de Swagger, ni de Bases de Datos. Son clases de Java puro.
  • Todas las anotaciones de documentación de Swagger (@Tag, @Operation, @Schema) se colocaron exclusivamente en la capa de Infraestructura (en el CourseController y en los DTOs CourseRequest y CourseResponse).

De esta manera, si mañana decides cambiar Swagger por otra herramienta, tu lógica de negocio y tu dominio ni se enteran. ¡Tu arquitectura está segura!

📖 2. ¿Qué se necesita para generar Swagger Automáticamente?

Históricamente se usaba una librería llamada SpringFox. Sin embargo, SpringFox dejó de actualizarse hace varios años y es incompatible con Spring Boot 3 (ya que Spring Boot 3 cambió internamente la paquetería de javax.* a jakarta.*).

La herramienta oficial, moderna y estándar hoy en día para lograr exactamente el mismo resultado (una interfaz Swagger UI) se llama Springdoc-OpenAPI.

Springdoc escanea automáticamente tus controladores (clases con @RestController) al iniciar la aplicación. Lee tus rutas (@GetMapping, @PostMapping) y tus parámetros, y construye de forma automática un documento JSON estándar llamado OpenAPI, que luego renderiza en la famosa página visual interactiva.

🚀 3. Pasos para implementarlo de forma automática

A continuación te detallo los pasos exactos que he seguido en tu código para que el proceso sea completamente automático:

1 Añadir la Dependencia:

En el archivo build.gradle, en el bloque de dependencias, agregamos la librería oficial. Esto le dice a Spring Boot que auto-configure todo el ecosistema de Swagger.

implementation 'org.springdoc:springdoc-openapi-starter-webmvc-ui:2.5.0'

2 Anotar el Controlador:

En tu CourseController, utilizamos la anotación @Tag para ponerle un título a la sección, y @Operation arriba de cada método para describir qué hace ese endpoint específico.

@Tag(name = "Courses", description = "Endpoints para gestionar cursos") public class CourseController { @GetMapping @Operation(summary = "Obtener cursos", description = "Lista todos los cursos...") public ResponseEntity<...> getCourses(...) { ... } }

3 Anotar los Modelos (DTOs):

Para que la documentación de Swagger le explique al frontend (Angular/React) qué campos enviar y recibir, decoramos los DTOs de infraestructura (CourseRequest, CourseResponse) con @Schema.

public class CourseRequest { @Schema(description = "Título del curso", example = "Curso de Java") private String title; }

4 ¡Arrancar la aplicación!

No tienes que hacer nada más. La magia de la autoconfiguración de Spring Boot hace que la interfaz gráfica se genere sola. Al darle "Play" a tu aplicación en IntelliJ, simplemente abres tu navegador web en:

http://localhost:8080/swagger-ui.html

Y verás toda tu API documentada, interactiva y lista para enviar pruebas (peticiones POST, GET) sin usar Postman.
